# Django 開発規約

Django プロジェクトにおける開発・運用のルールを定める。

## マイグレーション管理

### モデル変更時の必須手順

**モデルファイル (`models.py`) を変更した場合は、必ず以下の手順を実行すること。**

1. **マイグレーションファイルの生成**
   ```bash
   python manage.py makemigrations
   ```
   - 変更内容を検出し、マイグレーションファイルを自動生成する
   - 生成されたファイル名と内容を確認し、意図した変更が反映されているか検証する

2. **マイグレーションの適用**
   ```bash
   python manage.py migrate
   ```
   - データベースに変更を反映する
   - エラーが発生した場合は、原因を特定して修正する

3. **マイグレーションファイルのコミット**
   - 生成されたマイグレーションファイルは必ず Git にコミットすること
   - マイグレーションファイルとモデル変更は同一コミットに含めること

### マイグレーション実行のタイミング

以下の変更を行った場合は、必ずマイグレーションが必要:
- モデルクラスの追加・削除
- フィールドの追加・削除・変更（型、オプション、デフォルト値など）
- `db_table`, `db_column`, `unique_together`, `indexes` などのメタ情報の変更
- カスタムマイグレーション（データ移行、関数、トリガーなど）の追加

### マイグレーション確認コマンド

```bash
# 未適用のマイグレーションを確認
python manage.py showmigrations

# マイグレーションの状態を確認（どのマイグレーションが適用済みか）
python manage.py showmigrations --list

# マイグレーションの SQL を確認（実行前の検証用）
python manage.py sqlmigrate <app_name> <migration_number>
```

### 注意事項

- **マイグレーションファイルを手動で削除・編集しない**（特に本番環境に適用済みのもの）
- **複数人での開発時は、マイグレーションの競合に注意する**
  - `pull` 後は必ず `python manage.py migrate` を実行
  - マイグレーションファイルの番号が重複した場合は、`python manage.py makemigrations --merge` でマージする
- **本番環境へのデプロイ前に、必ずステージング環境でマイグレーションをテストする**

## その他の Django ルール

### 設定ファイルの管理

- 環境依存の設定（SECRET_KEY, データベース接続情報など）は環境変数または `.env` ファイルで管理する
- `.env` ファイルは `.gitignore` に追加し、リポジトリにコミットしない
- `.env.example` をテンプレートとして用意し、必要な環境変数を明示する

### 静的ファイルとメディアファイル

- 開発時は `python manage.py collectstatic` を必要に応じて実行
- 本番環境では静的ファイルを CDN または専用のストレージサービスで配信することを検討

### 管理コマンド

- カスタム管理コマンドは `management/commands/` ディレクトリに配置
- コマンド名は明確で、用途がわかりやすいものにする
- `--help` オプションで適切な説明が表示されるよう、docstring とヘルプテキストを記述する
