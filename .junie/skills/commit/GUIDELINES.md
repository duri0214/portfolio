# Junie 作業鉄則 (コミット & PR 管理ガイドライン)

今回の重大なインシデント（編集のみでコミット・PR作成を忘れる、ブランチ混在）を重く受け止め、以下の鉄則を策定しました。
Junie は作業を開始する際、および完了する際に、必ずこのガイドラインを自己チェックしなければなりません。

## 1. 作業開始時の鉄則 (Start-up)
- [ ] **現在のブランチを確認せよ**:
  - `git branch --show-current` で、自分がどのブランチにいるか確認する。
  - 大規模リファクタリングなど、他件の作業ブランチにいないか？
- [ ] **専用作業ブランチを即座に作成せよ**:
  - `master` から最新を取得し、`git checkout -b <type>/<issue-no>-<desc>` で新しいブランチを作成する。
  - **「ちょっとした修正だから」と master や他ブランチで直接編集することは厳禁。**

## 2. 作業中の鉄則 (In-progress)
- [ ] **コミット前にユーザーへ確認せよ**:
  - `git commit` を実行する前に、変更内容（`git diff`の結果）をユーザーに提示し、コミットの許可を得る。
- [ ] **論理的な単位でこまめにコミットせよ**:
  - ユーザーの許可が得られたら、`language="ja"` の追加など、一つの目的が達成された単位で即座にコミットする。
  - 編集だけしてセッションを終了することは、作業の放棄とみなす。
- [ ] **変更履歴 (git diff) を自己レビューせよ**:
  - コミット前に `git diff` で、意図しないファイルや行が混じっていないか最終確認する。

## 3. 作業終了時の鉄則 (Completion)
- [ ] **独断で Issue をクローズするな、マージを行うな**:
  - 作業が完了したと思っても、ユーザーによるマージが行われるまで、絶対に Issue をクローズしてはならない。
  - **Junie による PR のマージは厳禁。** ユーザーが GitHub 上でレビューし、手動でマージするのを待つこと。
  - ユーザーがマージを行えば、GitHub の Development 機能によって Issue は自動的に閉じられる。これが正解のフローである。
- [ ] **プッシュと PR 作成をセットで行え**:
  - コミットしたら必ず `git push` を行う。
  - **既存の PR がまだ作成されていない場合に限り**、`gh pr create` でプルリクエストを作成する。
  - すでに PR が存在する場合は、追加のプッシュによって PR 内容が更新されたことを確認し、その URL をユーザーに報告する。
- [ ] **Issue と PR を確実に紐付けよ**:
  - 新規に PR を作成する場合、GitHub の Development 機能（Linked Issues）を利用して対象の Issue を紐付ける。
  - **PR 本文に `Closes #<issue-no>` を記述することは禁止。**
- [ ] **PR URL をユーザーに報告し、レビューを待て**:
  - 修正報告には必ず PR の URL を添え、ユーザーによるレビューとマージを待つ。セッションを終了しても、Issue はオープンのままにすること。
  - **ユーザーの明示的な許可なく Issue をクローズすることは厳禁。**

## 4. インシデント発生時の対応
- もし手順を飛ばしてしまったことに気づいたら、即座に作業を止め、ユーザーに正直に報告し、正しいフロー（ブランチ切り出し、チェリーピック等）へ修正する。
