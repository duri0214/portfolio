# Value Object と Domain Model 詳細規約

Value Object や Domain Model を作成・変更するときだけ読む。

- Value Object は不変（Immutable）であり、それ自体が特定の意味や計算ロジックを持つべきである。
- Python における VO は、`dataclass` を用いて実装し、過度な抽象化や構造の変更を避けること。
- 実装のシンプルさ（KISS 原則）を優先する。
  - `frozen=True` や `dataclass` の機能を活用して不変性を保つことは推奨されるが、`__post_init__` で複雑な計算が必要になったり、`object.__setattr__` を使わなければならないような場合は、無理に適用しない。
  - `dataclass` を使うことでかえってコードが冗長になったり、直感的でなくなる場合は、プレーンなクラスと `__init__` メソッドを用いたシンプルな実装を優先すること。
  - 誰が見ても一目で意図が伝わり、メンテナンスしやすい素直なコードを目指す。
- VO の docstring には、必ず `Attributes` セクションを記述する。
  - 各フィールドの型と役割を明記し、IDE でマウスオーバー（ホバー）した際にその VO がどのようなデータを持っているか即座に判別できるようにすること。
  - 構造を複雑にせず、docstring の充実によって可読性と保守性を担保する。
- VO に外部の状態（DBの状態やグローバルなフラグなど）に依存する複雑な遷移ロジックを持たせるのは避ける。
- 状態遷移などの「振る舞い」が複雑になる場合は、Service や必要に応じて UseCase でそのフローを制御し、VO はあくまで「現在の状態」や「単純な次の候補」を示す役割に留める。
- VO 内で他のドメインサービスやリポジトリを呼び出すことは厳禁とする。
