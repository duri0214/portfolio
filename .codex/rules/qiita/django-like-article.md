# 編集ルール（スタイルガイド） — Django Like 2026

適用対象: docs/qiita/django_like_2026.md（このファイル専用）

目的:
- Qiita記事「Djangoでいいね！をつくる 2023リファクタリング版」を、Webアプリを一度作れるようになった人が次に実装したくなる機能として2026年版へ整理する。
- 元記事の「記事にいいねを付ける」素朴な流れを残しつつ、2026年時点のDjango実装として安全で読みやすい形に更新する。

適用スコープ:
- Django の投稿記事モデルに対するいいね機能
- ログインユーザーによるいいねの登録/解除
- Ajax/fetch による画面更新
- 非対象: SNS通知、ランキング、タイムライン、リアルタイム配信、レコメンド、汎用リアクション基盤

## ルール

1) 読者の入口を固定する
   - 「Webアプリをいったん作れるようになったあと、まず作りたくなる機能」として書く。
   - いいね機能は、投稿、ユーザー、ボタン、非同期通信、集計が小さくまとまった練習題材として扱う。
   - 大規模SNS設計や通知機能へ広げすぎない。

2) 最小構成を保つ
   - モデルは `Article` と `Like` の2つを基本にする。
   - `Like` は `user` と `article` の中間テーブルとして説明する。
   - 1ユーザーが同じ記事に複数回いいねできない制約を必ず入れる。

3) 認証の扱い
   - URLやリクエスト本文に `user_id` を渡さない。
   - いいねするユーザーは必ず `request.user` から取得する。
   - 未ログインの場合はログイン必須として扱い、勝手に匿名ユーザーのいいねを作らない。

4) トグル処理の説明
   - すでにいいね済みなら削除、未いいねなら作成する。
   - 返却値は最低限 `liked_by_me` と `likes_count` にする。
   - 処理後の再集計値を返し、フロント側で数字とボタン状態を更新する。

5) 一覧表示の説明
   - 記事一覧では、各記事のいいね数と自分がいいね済みかを同時に表示する。
   - `Count` と `Exists` / `OuterRef` を使う方針を優先し、ループ内で記事ごとにクエリを増やす例にしない。
   - 既存実装を紹介する場合も、N+1を避ける観点を添える。

6) CSRFとfetch
   - `fetch` のPOSTには `X-CSRFToken` を付ける。
   - HTMLに埋め込む場合は `{{ csrf_token }}` を使う前提にする。
   - APIをJSONで返す場合は `JsonResponse` を使う。

7) コードブロックの表記
   - Pythonファイルは `python:app/models.py` のように、コロン右側に対象ファイル名を付ける。
   - テンプレートは `html:app/templates/app/index.html` のように書く。
   - JavaScriptを分けて示す場合は `javascript:app/static/app/js/like.js` とする。
   - コンソール操作は `bash:console` を使う。

## チェックリスト（レビュー用）
- `user_id` をURLやPOST本文で受け取る例になっていないか。
- `Like` にユーザーと記事の一意制約があるか。
- 未ログイン時の扱いが明記されているか。
- CSRFトークン付きのPOST例になっているか。
- 一覧表示でいいね数と自分の状態をまとめて取得する説明になっているか。
- 元記事の素朴さを残しつつ、2026版として安全策を足せているか。
