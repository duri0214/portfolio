# ポートフォリオ共通業務ルール (Portfolio Common Rules)

## Git/GitHub 操作
- GitHub API、GitHub Contents API、MCP のファイル更新系ツールなどで、リモートブランチ上のファイルを直接作成・更新・削除しない。
- 変更は必ずローカル作業ツリーで編集し、`git add`、`git commit`、`git push` の通常フローで反映する。
- `git commit --amend` や `git push --force` / `git push --force-with-lease` による履歴上書きは原則禁止する。PR作成後の修正や追加対応は、新しいコミットとして履歴を残す。
- 履歴上書きが必要な場合は、ユーザーから明示的な許可を得て、理由と影響範囲を作業報告に残す。
- コード修正やルール・ドキュメント修正を行った場合は、必要な確認（フォーマット、関連テスト、差分確認など）を通したうえで、ユーザーが明示的に `commit不要`・`push不要`・`PR不要`・`まだコミットしない` と指示していない限り、確認待ちせずコミット、push、PR 作成まで進める。
- ユーザーが `commit` を明示した場合の詳細な確認手順や失敗時の扱いは、`commit` スキルに従う。
- ユーザーが `PR作って`、`pull request 作って` など PR 作成を明示した場合は、作成可能な状態なら本文提示だけで止めず、追加確認なしで `pull-request` スキルに従って PR 作成まで進める。
- `git add`、`git commit`、`git push`、`git checkout`、`git reset` などが失敗した場合は、API 等で迂回せず、原因を切り分けてユーザーに確認する。
- Issue/PR の作成、本文更新、コメント、メタ情報更新、状態確認などの GitHub 操作は、普段使っている `gh` CLI を使う。
- GitHub API、GitHub App、MCP の GitHub 更新系ツールで Issue/PR を作成・更新・コメントしない。権限差や挙動差でチケット重複・意図しない更新が起きるため。
- `gh` で GitHub 操作が失敗した場合も、API 等で迂回せず、原因を切り分けてユーザーに確認する。
- PRレビューコメントの指摘を修正した場合は、PR全体への新規コメントではなく、原則として元のレビューコメントに対する返信として `gh` でコメントし、どのコミット/変更で対応したかを簡潔に残す。
- 元のレビューコメントへ返信する場合は、GitHub上で指摘との関連が分かるよう、REST API の `pulls/comments/{comment_id}/replies` など、`in_reply_to_id` が元コメントを指す方法を使う。元コメントが outdated 等で返信できない場合だけ、対象コメントURLを明記してPRへコメントする。

## GitHub Issue/PR ラベル運用
- 対象アプリが明確な Issue/PR には、種別・優先度ラベルとは別に `app: <slug>` 形式のアプリラベルを付ける。
- アプリラベルは、主な変更対象となる Django アプリ、共通ライブラリ、または機能領域を示す。複数領域に同程度の変更が入る場合だけ複数のアプリラベルを付ける。
- 横断的な設定、デプロイ、CI、ドキュメント、リポジトリ運用だけの Issue/PR には、無理にアプリラベルを付けない。
- ラベル名は GitHub 上に存在するものを優先し、新規作成が必要な場合は既存の `app: ...` 命名に合わせる。
- `app:` ラベルが存在していても、Issue への紐づきが 0 件のものは対応表へ常設しない。必要になった時点で対応表へ追加する。
- app ラベルを判断するときだけ、`.codex/rules/references/app-labels.md` の対応表を読む。

## バッチ/管理コマンド命名
- Django 管理コマンド、cron、seed、fixture、初期データ投入を作成・変更するときだけ、`.codex/rules/references/management-commands.md` を読む。

## 設計方針
- 基本的に DDD（ドメイン駆動設計）のエッセンスに沿って設計・実装する。
- `domain/` 配下は基本的に Repository、Service、Value Object の3層で構成する。
- 業務ルールや判断ロジックは、可能な限り `domain/` 配下の Value Object、Service、Repository に集約する。
- Django の `views.py`、`forms.py`、`models.py` はフレームワーク連携や入出力、永続化の責務を中心にし、業務ロジックを過度に持たせない。
- Repository は永続化の詳細を隠蔽し、Service が Django モデルのクエリ操作に直接依存しないようにする。
- UseCase、Factory など追加の DDD 構造は、必要性が明確な場合にだけ作る。
- 基本の3層で十分な場合は追加の層を作らず、過剰品質を避ける。

## UI/CSS 方針
- 画面実装は基本的に Bootstrap の既存コンポーネント、ユーティリティクラス、グリッドをベースにする。
- 独自 CSS は、Bootstrap だけでは表現しにくい情報設計、状態表現、細かなレイアウト調整に限定し、最小限にする。
- CSS を使わないという意味ではない。まず Bootstrap で作り、必要な差分だけをアプリ固有 CSS に書く。

## UI 権限制御方針
- 「スーパーユーザーのみ」「管理者のみ」など、特定権限だけが実行できる操作を画面に置く場合、権限のないユーザーに対してボタンや導線を消さない。
- 権限のないユーザーには、原則として同じ位置に disabled 状態のボタンを表示し、機能の存在と実行不可であることを分かるようにする。
- disabled 状態の権限制御ボタンには、なぜクリックできないのか、どうすれば使えるようになるのかを近くに表示する。例: 「管理者権限が必要なボタンは無効化されています」「管理者権限が必要なため、データ取得ボタンは無効化されています」。
- ただし、秘匿すべき管理機能、セキュリティ上存在を見せるべきでない操作、またはユーザー体験として明確に不要な導線は例外とし、その理由を実装またはPR説明に残す。

## 外部データ表示の透明性
- e-Stat、EDINET、Yahoo API、官公庁統計、CSV取り込みなど、外部データを画面・レポート・ダッシュボードに表示するときだけ、`.codex/rules/references/external-data-display.md` を読む。

## ユーザー管理の設計パターンの統一
- すべてのアプリにおいて、認証（ID/Pass）は Django 標準の `auth.User` を使用し、アプリ固有の属性やロール（役割）の管理は、各アプリ内で `User` と 1:1 で紐づく `UserAttribute` モデル（またはそれに相当するプロフィールモデル）を定義して行う。
- 同一の `User` に対し、アプリごとに異なる役割（例：`hospital` では医師/患者、`shopping` ではスタッフ/購入者）を独立して定義できる構成を維持すること。
- これにより、システム全体の認証基盤を共通化しつつ、各ドメインに特化した柔軟なユーザー属性管理を実現する。
