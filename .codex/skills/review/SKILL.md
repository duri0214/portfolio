# Review

## 概要

このスキルは、このリポジトリでPRレビューやレビューコメント対応を行うための手順をまとめる。現在のPRに紐づけて確認し、GitHubへアクセスできる場合は、対応したインラインコメントへ追跡できる返信を残す。

## 手順

1. レビュー対象のPRを特定する。
   - ユーザーがPR番号やURLを指定している場合はそれを使う。
   - 指定がない場合は、現在のブランチから `gh pr view --json number,url` で確認する。
   - `gh` の認証が無効な場合は、GitHubの読み書きを止めて `gh auth login -h github.com` が必要だと伝える。
2. レビューコメントをスレッド単位で確認する。
   - 未解決レビュー、インラインコメント、requested changes が関係する場合は `github:gh-address-comments` を使う。
   - 対応が必要なコメントを、ファイル別または挙動別にまとめる。
   - 解決済み、重複、情報共有のみのコメントは、ユーザーが求めていない限り対応対象から外す。
3. レイヤーの棲み分けを確認する。
   - Repository: 永続化、クエリ、外部ストレージや外部APIの集合アクセスを担当する。
   - Service: 処理手順やドメイン操作を担当する。Repository境界がある場合は、Service内で場当たり的にDjangoモデルを直接クエリしない。
   - Value Object: dataclass や値の意味を持つ型を置く。新しいValue ObjectをServiceやRepositoryファイルへ混在させない。
4. ローカルで修正する。
   - 各修正がどのレビューコメントに対応するか追えるようにする。
   - 関係ないリファクタリングへ広げない。
   - 初期実装や試行錯誤で入れた残骸を残さず、最終的に必要な実装だけへ整理する。モデル修正を伴う仮実装の後始末には、migration の作成・削除・適用確認まで含める。
   - KISS と YAGNI を意識し、後方互換や抽象化を「今回PR内で消えた実装」のためだけに残さない。
   - 既に作業ツリーにあるユーザー変更は壊さない。
5. 確認する。
   - 変更したアプリや挙動に絞ってテストを実行する。
   - テストを実行できない場合は、理由を報告する。
6. ユーザーがコメント返しを求めている場合は、対応済みコメントへ返信する。
   - 原則として、元のインラインレビューコメントに `gh` で返信する。
   - 可能なら `in_reply_to_id` が元コメントを指すスレッド返信にする。
   - コメントが outdated などで直接返信できない場合だけ、元コメントURLと対応内容をPRコメントに書く。

## レビュー結果の書き方

レビューとして回答するときは、重大度順に指摘を先に書き、ファイルと行番号を添える。問題が見つからない場合は明確にそう書き、残るテスト不足やリスクがあれば短く補足する。
