# コミット・push・PR連携

コミットを作成する前に、ルール適合確認、今回のブランチで変更された対象へのフォーマッター、Django のテストを必ず実行する。コミットと push は、それぞれ実行前に UI でユーザーへ確認する。

## 手順

1. 現在のブランチと変更内容を確認する。
   ```bash
   git status --short --branch
   ```
2. `master` ブランチ上にいる場合はコミットせず、トピックブランチへの切り替えを促す。
3. `AGENTS.md` と `.codex/rules/` 配下の関連ルールを読み、変更内容が一通りルールに従っているか確認する。
4. ルール違反が見つかった場合は、今回の変更に関係する範囲で修正してから次へ進む。
5. フォーマッターは、今回のブランチで変更が入った対象に限定して実行する。明示的に「全ソースコードをフォーマットして」と指示された場合のみ全体へ実行する。
   ```bash
   black <変更されたPythonファイル>
   ```
6. Django のテストを実行する。
   ```bash
   python manage.py test
   ```
7. フォーマットで差分が増えた場合は、今回の変更対象に含まれる差分だけを確認してコミット対象に含める。
8. ルール適合確認、フォーマッター、テストが成功したら、差分と実行結果を要約して、コミットしてよいか UI でユーザーに確認する。
9. ユーザーが許可した場合のみ、コミットを作成する。
10. コミット作成後、push してよいか UI でユーザーに確認する。
11. ユーザーが許可した場合のみ、現在のブランチを push する。
12. push 後、`.codex/skills/pull-request/SKILL.md` を読み、pull-request スキルのルールに引き継いで PR 本文作成または PR 作成を進める。

## 失敗時

- `black <変更されたPythonファイル>` が失敗した場合は、原因を修正してから再実行する。
- `python manage.py test` が失敗した場合は、失敗内容を確認し、今回の変更に起因する問題だけを修正して再実行する。
- ルール違反が既存の無関係な内容に由来する場合は、内容をユーザーへ報告し、今回のコミットで扱うか確認する。
- 既存の無関係な失敗だと判断できる場合は、失敗内容をユーザーへ報告し、コミットしてよいか確認する。
- コミットまたは push の UI 確認でユーザーが許可しない場合は、その時点で停止する。

## コミットメッセージ

- 変更内容が分かる短いメッセージにする。
- Issue 番号がブランチ名から分かる場合は、必要に応じて含める。
