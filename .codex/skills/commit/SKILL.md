# コミット・push・PR連携

コミットを作成する前に、ルール適合確認、今回のブランチで変更された対象へのフォーマッター、関連範囲の Django テストを必ず実行する。ユーザーが commit スキルを明示した場合は、その依頼をコミット許可として扱い、チェック成功後に追加確認なしでコミットを作成する。コミット作成後は、ユーザーが明示的に push 不要と指示していない限り、基本的に現在のブランチを push してよい。push 後に PR 作成まで進めるかは `.codex/rules/portfolio.md` の通常フローとユーザーの明示的な停止指示に従う。ただし、マージ済みブランチに追加コミットしてはいけない。

## 手順

1. 現在のブランチと変更内容を確認する。
   ```bash
   git status --short --branch
   ```
2. `master` ブランチ上にいる場合はコミットせず、トピックブランチへの切り替えを促す。
3. 現在のブランチがすでにマージ済みでないか確認する。PR がある場合は `gh pr view --json state,mergedAt,baseRefName` などで merged 状態を確認し、PR がない場合も必要に応じて `git branch --merged` で確認する。
4. 現在のブランチがマージ済みだった場合は、そのブランチにはコミットしない。新しいトピックブランチを作るか、どのブランチで作業するかをユーザーに確認する。
5. `AGENTS.md` と `.codex/rules/` 配下の関連ルールを読み、変更内容が一通りルールに従っているか確認する。
6. ルール違反が見つかった場合は、今回の変更に関係する範囲で修正してから次へ進む。
7. フォーマッターは、今回のブランチで変更が入った対象に限定して実行する。明示的に「全ソースコードをフォーマットして」と指示された場合のみ全体へ実行する。
   ```bash
   black <変更されたPythonファイル>
   ```
8. Django の関連テストを実行する。毎回フルテストは行わず、変更範囲から十分に広めの対象を選ぶ。
   - アプリ配下の変更: 原則 `python manage.py test <app_name>` を実行する。
   - 特定機能の変更: まず該当テストモジュールや近いテストを実行し、必要に応じてアプリ全体へ広げる。
   - 共通ライブラリ、設定、モデル、マイグレーション、複数アプリ横断、影響範囲が読みにくい変更: `python manage.py test` へ昇格する。
   - スキルやドキュメントのみの変更: 関連する検証コマンドがある場合はそれを実行し、Django テストは省略してよい。
   - どのテストを選んだか、なぜ十分と判断したかをコミット前の要約に含める。
   ```bash
   python manage.py test <related_target>
   ```
9. フォーマットで差分が増えた場合は、今回の変更対象に含まれる差分だけを確認してコミット対象に含める。
10. ルール適合確認、フォーマッター、テストが成功したら、差分と実行結果を簡潔に記録する。
11. ユーザーが明示的に commit 不要またはまだコミットしないと指示していない限り、追加確認なしでコミットを作成する。変更範囲やコミット対象を一意に判断できない場合だけ、必要な不足情報を確認する。
12. コミット作成後、ユーザーが明示的に push 不要と指示していない限り、現在のブランチを push する。
13. push 後、ユーザーが明示的に PR 不要と指示していない限り、`.codex/skills/pull-request/SKILL.md` を読み、pull-request スキルのルールに引き継いで PR 作成まで進める。

## 失敗時

- `black <変更されたPythonファイル>` が失敗した場合は、原因を修正してから再実行する。
- 関連テストが失敗した場合は、失敗内容を確認し、今回の変更に起因する問題だけを修正して再実行する。
- 関連テストだけでは不安が残る失敗や影響範囲が広がった場合は、より広いテスト対象または `python manage.py test` へ昇格する。
- ルール違反が既存の無関係な内容に由来する場合は、内容をユーザーへ報告し、今回のコミットで扱うか確認する。
- 既存の無関係な失敗だと判断できる場合は、失敗内容をユーザーへ報告し、コミットしてよいか確認する。
- ユーザーが commit 不要またはまだコミットしないと指示している場合は、その時点で停止する。
- コミット前に現在のブランチがマージ済みだと分かった場合は、コミットせず、作業先ブランチをユーザーに確認する。
- push が失敗した場合は、API 等で迂回せず、原因を切り分けてユーザーに確認する。

## コミットメッセージ

- 変更内容が分かる短いメッセージにする。
- Issue 番号がブランチ名から分かる場合は、必要に応じて含める。
