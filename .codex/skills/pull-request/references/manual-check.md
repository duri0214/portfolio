# 目検による動作確認手順

## 判断基準

- 既存または追加したAPIテストで担保できる挙動は、目検手順に長い確認コマンドとして再実装しない。
- APIテストで担保済みの内容は、`## 自動テストでカバーできた範囲` にテスト名や確認観点を書く。
- 目検手順は、テストで担保しにくい画面表示、外部連携、環境差分、手元での最小スモーク確認に絞る。
- 目検対象がない場合は、無理にコマンドを作らず「目検対象なし。自動テストでカバー済み」と明記する。

## コマンドを書く場合

- PR本文へ載せる前に一通り実行し、成功したコマンド、ステータス、状態変化を本文へ反映する。
- 失敗したコマンドは未確認手順として残さず、原因を直してから本文を更新する。
- 出力はレビュアーが要点を拾いやすい形にし、巨大なオブジェクトやネスト値を省略表示のまま出さない。
- エラー時は原因が分かるように明確に止める。HTMLエラーレスポンスなどをJSON変換して二次エラーにしない。

## DB状態に依存する場合

- レビュアーが再実行しても同じ前提になるよう、開発・検証用DBのリセット、fixture再投入、確認用データ作成を明記する。
- 本番DBや共有DBで実行してはいけない前提を必要に応じて明記する。
- 実DBの既存ID、前回の目検で残ったデータ、手元環境だけに存在するレコードには依存しない。
- DBリセットを行わない場合は、理由と確認開始時に必要な初期データ、数量、IDの作成または確認手順を書く。
