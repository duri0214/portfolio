# チケット・Issue 作成規約

- **出力言語は必ず日本語とすること。**
- 会話の流れと調査結果から、GitHub Issue に起票できる作業内容を整理する。
- ユーザーから「issue書いて」「チケット作って」などの依頼があった場合は、`gh issue create` で GitHub Issue を作成する。
- ユーザーから明示的に「本文だけ」「下書きだけ」と依頼された場合のみ、Markdown のコードブロック（生md）で Issue 本文を生成する。
- Issue 作成依頼は作成許可として扱い、作成内容とメタ情報を一意に判断できる場合は追加確認なしで `gh issue create` まで実行する。
- 親チケットは、現在ブランチ名の先頭番号、現在ブランチの open PR、または会話で明示された Issue から推定する。推定元 Issue に親チケットがある場合は、その親を新規 Issue の親候補として扱う。現在ブランチが `master` / `main` の場合、または現在ブランチに紐づく Issue/PR が closed/merged の場合は作業中の親候補とは扱わず、新規チケットとして作成する。
- 会話の流れ、現在ブランチ、関連 Issue/PR、変更予定ファイルから対象アプリを推定できる場合は、確認なしで既存の `app: <slug>` ラベルを付ける。推定が外れていた場合はユーザーが後で直せるため、作成後の報告で付けたラベルを明記する。

## Issue 本文の生成フォーマット

```markdown
**タイトル:** [タイトル]

**本文:**
## 概要
[概要]

## 主な対応内容
- [対応内容]

## 期待される効果
- [効果]
```

## 報告

- Issue 本文だけを生成した場合は、本文だけ生成したことを簡潔に報告する。
- GitHub Issue を作成した場合は、作成した Issue のURLと要約を報告する。
- 親チケットを指定された場合は、親子関係を設定できたか報告する。
- app ラベルを付けた場合は、どのラベルを付けたか報告する。推定できず付けなかった場合も、その旨を短く報告する。
- Issue 作成後にブランチを作る場合は、`branch` スキルへ引き継ぐ。

## Issue 作成時のメタ情報設定

GitHub Issue を実際に作成する場合は、本文生成に加えて以下をできるだけ設定すること。設定できない項目があっても、Issue 作成自体は止めないこと。

- **Parent issue**: 現在ブランチ名の先頭番号、現在ブランチの open PR、または会話で明示された Issue を確認し、その Issue に親チケットがある場合は新規 Issue の親として扱う。現在ブランチが `master` / `main` の場合、または現在ブランチに紐づく Issue/PR が closed/merged の場合は、そこから親を引き継がず親なしの新規チケットとして扱う。親候補を一意に決められる場合は追加確認なしで本文に `親チケット: #<番号>` を含め、Issue 作成後に GitHub の Sub-issues 機能で親子関係を設定する。親候補がない場合は親なしで作成する。親候補が複数あり一意に決められない場合だけユーザーに確認する。設定できない場合は、本文上の参照だけは残し、失敗理由を報告する。
- **Assignee**: `gh api user --jq .login` などで自分自身の GitHub ユーザー名を取得し、Issue の assignee に自分自身を設定する。
- **Label**: 依頼内容や対象アプリから適切な label を推測できる場合は確認なしで設定する。アプリラベルは `.codex/rules/portfolio.md` の「GitHub Issue/PR ラベル運用」を参照し、既存の `app: <slug>` ラベルを優先する。判断できない場合は省略する。ラベルを付けた場合も省略した場合も、作成後の報告に含める。
- **Projects**: `gh project list --owner duri0214 --format json` などで Project 一覧を取得し、取得できた Project が1件だけなら、その Project の `title` を使って Issue に設定する。Project 名を固定文字列で決め打ちしないこと。Project 操作の権限が不足している場合は、`gh auth refresh -s read:project -s project` が必要なことを伝える。

## 確認が必要な場合

- Issue の目的、タイトル、本文を会話や調査結果から一意に作れない場合。
- 親候補が複数あり、どれを親にするべきか一意に判断できない場合。
- ユーザーが「本文だけ」「下書きだけ」と明示している場合。
- `gh issue create` やメタ情報設定が失敗し、API 等で迂回せず原因切り分けが必要な場合。
