# ブランチ運用ルール

プロジェクトの品質向上と円滑な開発のため、以下のブランチ運用ルールを遵守してください。

## 基本方針

- ユーザーが `branch` スキルを明示した場合や、ブランチ作成を依頼した場合は、その依頼をブランチ作成許可として扱う。
- 作業に対応する Issue があり、現在の作業ツリーがブランチ作成可能な状態なら、追加確認なしでトピックブランチを作成して切り替える。
- ブランチ作成後の Issue assignee 設定、Project 追加、Project ステータス更新も、設定対象を一意に判断できる場合は追加確認なしで実行する。
- 確認が必要な場合は、ブランチ名や操作自体の承認ではなく、判断に必要な不足情報だけを短く確認する。

## 1. ブランチ運用のルール

- **作業開始前の Issue 必須**
    - すべての開発タスクは、着手前に対応する GitHub Issue を用意してください。
    - 対応 Issue がない場合は、先に `ticket` スキルで Issue を作成してからブランチ作成へ進んでください。
    - ユーザーが明示的に Issue なしで進めるよう依頼した場合だけ例外とし、例外であることを作業報告に明記してください。
- **既存ブランチの再利用可否を確認**
    - Issue 番号付きブランチにいる場合でも、作業開始前に `git branch --merged` や PR/Issue の状態を確認してください。
    - 現在のブランチが既に `master` へマージ済み、または対応 Issue/PR が完了済みの場合、そのブランチに新しい変更を載せないでください。
    - マージ済みブランチ上で未コミット変更を作ってしまった場合は、変更を退避し、最新の `master` から新しい Issue 番号付きブランチを作成して差分を載せ替えてください。
- **master ブランチへの直接コミット禁止**
    - `master` ブランチにいるときは、直接コミットを行わないでください。
    - すべての変更は必ずトピックブランチを作成してから行ってください。
- **ブランチ作成後の Issue ステータス更新**
    - トピックブランチ名は、対応する Issue 番号を先頭に付けて作成してください（例：`123-feature-name`）。
    - ブランチを作成して切り替えたら、該当 Issue の Project ステータスを `Todo` から `In progress` に更新してください。
    - 該当 Issue が親チケットを持ち、親チケットの Project ステータスが `Todo` の場合は、初の子チケット着手とみなして親チケットも `In progress` に更新してください。
    - 該当 Issue が Project に未登録の場合は、`ticket` スキルの Issue 作成時メタ情報設定を確認してください。Project 一覧を取得でき、追加先を 1 件に特定できる場合は、追加確認なしでその Project に追加して `In progress` へ更新してください。
    - Project ステータス操作の権限が不足している場合は、`gh auth refresh -s read:project -s project` が必要なことを伝えてください。

## 2. ブランチ作成手順

1. `git status --short --branch` で現在ブランチと作業ツリーを確認する。
2. 現在ブランチが Issue 番号付きブランチの場合でも、`git branch --merged` や `gh pr view --json state,mergedAt,baseRefName` でマージ済みでないか確認する。
3. 作業ツリーに未コミット変更がなく、現在ブランチがマージ済みまたは `master` の場合は、最新の `origin/master` を取得してそこから新しいトピックブランチを作成する。
4. 作業ツリーに未コミット変更がある場合は、変更内容を壊さないように扱う。マージ済みブランチ上の変更なら退避してから最新 `master` 起点の新ブランチへ載せ替える。
5. ブランチ名は `<Issue番号>-<短い説明>` とし、説明部分は英小文字・数字・ハイフンで簡潔にする。
6. ブランチ作成後、Issue の assignee、Project、Project ステータスを設定できる範囲で更新する。

## 3. Issue 作成からの引き継ぎ

- 開発に着手する前に「issue書いて」「チケット作って」「ticket作って」などの指示があった場合は、`ticket` スキルを使用してください。
- `ticket` スキルで GitHub Issue 作成まで完了した後に、Issue 番号を使ってトピックブランチ作成へ進んでください。

## 4. 確認が必要な場合

- 対応 Issue がなく、ユーザーが Issue なしで進める例外を明示していない場合。
- 複数の Issue や Project が候補になり、作業対象を一意に決められない場合。
- 未コミット変更があり、退避・載せ替え・コミットのどれを選ぶべきか判断できない場合。
- 現在のブランチや Issue/PR の状態が矛盾していて、続行すると既存作業を壊す可能性がある場合。
- `gh` や `git` の操作が失敗し、API 等で迂回せず原因切り分けが必要な場合。
