# ブランチ・チケット運用ルール

プロジェクトの品質向上と円滑な開発のため、以下のブランチおよびチケット作成ルールを遵守してください。

## 1. ブランチ運用のルール

- **master ブランチへの直接コミット禁止**
    - `master` ブランチにいるときは、直接コミットを行わないでください。
    - すべての変更は必ずトピックブランチを作成してから行ってください。

## 2. チケット作成のルール

開発に着手する前に「チケット（issue）を書いてください」と指示があった場合、以下の手順で進めてください。

1. **修正予定のドキュメント化**
    - これまでの会話の流れから、どのような修正を予定しているのかを整理します。
    - `docs/tickets/` ディレクトリ内に、GitHub Issue を作成するためのチケット案（Markdown 形式）を作成します。
    - ファイル名は内容が分かるものにしてください（例：`docs/tickets/feature_xxx_issue.md`）。

2. **チケット案の内容**
    - 現状の課題
    - 調査結果（あれば）
    - 修正方針
    - タスク一覧
    - 完了条件

3. **報告**
    - チケット案を作成したら、その内容を提示し、GitHub Issue を作成する準備ができたことを報告してください。
    - その後、GitHub上でブランチを作成し、ローカルでそのブランチを `git pull` して切り替えるようユーザーに促してください。
